# コンテンツ拡張設計(動画ライブラリ・愛用ガジェット・FAQ)

視聴率・リーチ向上を目的としたポータルサイトのコンテンツ拡張の設計ドキュメント。

## 1. 市場調査サマリ

クリエイター公式サイトの定番コンテンツと SEO 施策を調査した結果:

- **公式サイトは「チャンネル外の信頼拠点」**。YouTube / SNS だけの発信はアルゴリズム変動に弱く、
  ファンの受け皿・案件窓口・検索流入の起点として独自サイトのコンテンツ充実が有効。
- **トピック別のコンテンツ整理(クラスタリング)と動画⇔サイトの相互リンク**が Google / YouTube
  双方の評価向上に寄与する。動画をカテゴリ別に整理したアーカイブは検索流入と回遊の両方に効く。
- **「使用機材・愛用ガジェット」ページはテック系チャンネルの定番**であり、視聴者の関心が高く
  アフィリエイト収益にも直結する。当チャンネルは動画概要欄に機材一覧を掲載済みで、素材が揃っている。
- **FAQ / お仕事窓口**は視聴者・案件依頼者の双方に価値があり、問い合わせ導線(マシュマロ・メール)を
  明確化できる。
- **法令対応**: 2023 年 10 月施行のステマ規制(景品表示法)により、アフィリエイトリンクを含む
  ページには「アフィリエイト広告を含む」旨をファーストビュー等の分かりやすい位置に明示する必要がある。
  また Google はアフィリエイトリンクへの `rel="sponsored"` 付与を推奨している。

チャンネルの発信テーマは「ITで日常をより便利に」(AI・ガジェット/家電・社会人 Vlog・
エンジニアキャリア)。拡張コンテンツはすべてこのテーマに沿って構成する。

## 2. 要件定義

### 2.1 追加ページ

| ページ | パス | 目的 | KPI への寄与 |
| --- | --- | --- | --- |
| 動画ライブラリ | `/videos/` | 全動画をカテゴリ・キーワードで探せるアーカイブ | 過去動画への回遊・検索流入・総再生時間 |
| 愛用ガジェット | `/gear/` | 動画で使用している機材・ガジェットの一覧 | 検索流入(機材名)・アフィリエイト収益・信頼感 |
| よくある質問 | `/faq/` | チャンネル・運営者・機材・お仕事依頼の FAQ | 問い合わせ導線の明確化・指名検索の受け皿 |

### 2.2 機能要件

- **FR-1**: 全ページ共通のヘッダーナビゲーション(ホーム / 動画ライブラリ / 愛用ガジェット / FAQ)と
  共通フッターを設ける。現在ページは `aria-current="page"` で示す。
- **FR-2**: `/videos/` は `videos.json` の全動画を新しい順に表示し、
  タイトルから自動分類したカテゴリ(AI・テック / ガジェット・家電 / Vlog・日常 / キャリア・学び)で
  絞り込みできる。キーワード検索(タイトル部分一致)と件数表示を備える。
- **FR-3**: 絞り込み・検索は JavaScript 無効時でも全件表示にフォールバックする
  (コンテンツは全件静的生成し、JS は表示/非表示の切替のみ行う)。
- **FR-4**: `/gear/` は `src/data/gear.json`(手動管理)からカテゴリ別に機材を表示する。
  ページ冒頭にアフィリエイト広告を含む旨を明示し、外部リンクには `rel="sponsored"` を付与する。
- **FR-5**: `/faq/` は `src/data/faq.json`(手動管理)からカテゴリ別に Q&A を表示し、
  `FAQPage` の JSON-LD を出力する。質問への導線(マシュマロ)とお仕事依頼窓口(メール)を含む。
- **FR-6**: 追加ページはサイトマップに自動掲載され(`@astrojs/sitemap`)、
  個別の title / description / canonical / OGP と `BreadcrumbList` JSON-LD を持つ。

### 2.3 非機能要件

- 既存方針を踏襲: 静的生成・JS 最小限・`lang="ja"`・WCAG 配慮(キーボード操作・aria 属性・コントラスト)。
- Lighthouse CI のゲート(Performance 0.85 / Accessibility 0.95 / Best Practices 0.95 / SEO 1.0)を
  全ページで満たす。
- データ層は既存の `parseVideosData` と同様、スキーマ検証付きパーサを設けテストする。

### 2.4 スコープ外(将来拡張)

- ブログ / 記事コンテンツ(執筆運用が必要なため)
- 動画個別ページ(RSS から得られる情報が薄く、YouTube と重複コンテンツになるリスクがあるため)
- メールマガジン・お問い合わせフォーム(静的サイトの範囲を超えるため)

## 4. UI / UX 設計

### 4.1 共通ナビゲーション

- ヘッダー: 上部固定(sticky)。左にサイト名(ホームへ)、右に「動画ライブラリ / 愛用ガジェット / FAQ」。
  リンクは 3 件のみのためハンバーガーメニューは設けず、モバイルでは横スクロール可能な 1 行に収める(JS 不要)。
- フッター: サイト内リンク + YouTube / X リンク + コピーライト。全ページで共通化(`Base.astro` に統合)。
- 既存トップページのヒーロー・セクション構成は変更しない(ヘッダーが 1 本増えるのみ)。

### 4.2 動画ライブラリ(`/videos/`)

- ページ見出し + 総件数 → フィルタ UI → 動画グリッド(レスポンシブ 1〜3 列、Shorts も同一グリッドで
  16:9 サムネイル表示に統一し、カードに「Shorts」バッジを付ける)。
- フィルタ UI:
  - カテゴリチップ(すべて / AI・テック / ガジェット・家電 / Vlog・日常 / キャリア・学び)。
    `<button aria-pressed>` で実装し、選択状態を視覚とスクリーンリーダー両方に伝える。
  - キーワード検索 `<input type="search">`(ラベル付き)。
  - 表示件数は `aria-live="polite"` の領域で通知。0 件時は空状態メッセージを表示。
- カードはトップページと同トーン(角丸・ホバーで浮く)。クリックで YouTube の動画ページを新規タブで開く。

### 4.3 愛用ガジェット(`/gear/`)

- ページ冒頭に PR 表記(アフィリエイト広告を含む旨)を目立つ位置に表示(ステマ規制対応)。
- カテゴリ(デスク環境 / 撮影・配信機材 / オーディオ・スマート家電)ごとにセクション分割し、
  各アイテムをカード(商品画像・製品名・ブランド・ひとことコメント・外部リンク)で表示。
- ページ冒頭に Amazon Storefront への導線を表示する。商品画像は Amazon またはメーカー公式の商品ページ由来の URL を利用する。
- Storefront 導線の直下に看板を持つ動物キャラクターを配置し、ページ訪問時に下から弾むように登場させる。`prefers-reduced-motion` ではアニメーションを無効化する。
- 冒頭の Storefront マスコットが画面上へ完全に外れた後は同じキャラクターを右下へフローティング表示し、Storefront へのリンクとして機能させる。上へ戻って冒頭のマスコットが再び見えたら非表示にする。
- 商品画像は 4:3 の固定枠内にアスペクト比を保って表示し、画像本体の上限を幅 288px・高さ 224px とする。
- 外部リンクは `target="_blank" rel="sponsored noopener noreferrer"`。

### 4.4 よくある質問(`/faq/`)

- カテゴリ(チャンネルについて / 運営者について / 機材・環境 / お仕事のご依頼)ごとに
  `<details>/<summary>` のアコーディオンで表示(JS 不要・キーボード操作可)。
- 「質問を送る」CTA(マシュマロ)をページ末尾に配置。

## 5. 外部設計

### 5.1 URL / ページ構成

```
/            トップ(既存。ヘッダー/フッター共通化のみ)
/videos/     動画ライブラリ
/gear/       愛用ガジェット
/faq/        よくある質問
```

- ビルドは Astro の directory 形式(`/videos/index.html`)。canonical・sitemap は自動整合。
- 各ページの meta: title「動画ライブラリ | MayaBase」等、description は個別に記述。

### 5.2 構造化データ(JSON-LD)

| ページ | 出力 |
| --- | --- |
| `/videos/` | `BreadcrumbList` + `ItemList`(全動画の `VideoObject`) |
| `/gear/` | `BreadcrumbList` + `ItemList`(`Product` は価格情報が無いため出力しない。名称のみの `ListItem`) |
| `/faq/` | `BreadcrumbList` + `FAQPage`(`Question` / `Answer`) |

※ FAQ リッチリザルトは 2023 年以降ほぼ表示されないが、セマンティクスとして valid な
`FAQPage` を出力しておく(コストゼロ・減点なし)。

### 5.3 データファイル

- `src/data/gear.json`(手動管理): 動画概要欄の「使用ガジェット・デバイス一覧」を初期データとして整備。
- `src/data/faq.json`(手動管理): 公開情報(プロフィール・概要欄・マシュマロ)に基づく Q&A。
- `src/data/videos.json`(既存・自動更新): 変更なし。カテゴリはビルド時にタイトルから導出する
  (JSON にカテゴリを持たせると RSS 自動更新と競合するため保存しない)。

## 6. 内部設計

### 6.1 モジュール構成

```
src/
├── lib/
│   ├── categories.ts    # 動画カテゴリの型・定義・自動分類ロジック
│   ├── gear.ts          # gear.json のスキーマ検証パーサ + カテゴリ定義
│   └── faq.ts           # faq.json のスキーマ検証パーサ
├── components/
│   ├── SiteHeader.astro # 共通ヘッダー(ナビ)
│   ├── SiteFooter.astro # 共通フッター
│   └── ArchiveCard.astro# 動画ライブラリ用グリッドカード(Shorts バッジ対応)
├── pages/
│   ├── videos.astro
│   ├── gear.astro
│   └── faq.astro
└── layouts/Base.astro   # ヘッダー/フッターを統合(全ページ共通化)
```

### 6.2 カテゴリ自動分類(`categories.ts`)

- カテゴリ: `ai`(AI・テック)/ `gadget`(ガジェット・家電)/ `vlog`(Vlog・日常)/
  `career`(キャリア・学び)/ `other`(その他)。
- **タイトルのみ**をキーワード照合の対象とする。概要欄はハッシュタグ(#エンジニア #vlog 等)が
  全動画に付いており誤分類の原因になるため使わない。
- 照合の優先順位: `career` → `gadget` → `ai` → `vlog` → `other`。
  (例: 「働きながら **AI 大学院**を卒業」は career、「**GPT-Live**」は ai に分類するため、
  より特異度の高いキーワード群から順に判定する)
- 1 動画 1 カテゴリ(主カテゴリ)。UI の絞り込みが直感的になることを優先する。
- キーワードは定数配列で管理し、変更はテストで担保する。

### 6.3 絞り込み・検索(クライアントサイド)

- 全カードを静的生成し、`<li>` に `data-category` と `data-title`(小文字化済み)を付与。
- ページ内 `<script>`(Astro がバンドル)でチップ選択・入力に応じて `hidden` 属性を切替。
- 状態は DOM のみで管理(カテゴリ 1 つ + 検索文字列)。URL 同期・ライブラリ導入はしない(JS 最小限)。

### 6.4 データスキーマ

```ts
// gear.json
interface GearData {
  items: GearItem[];
}
interface GearItem {
  name: string;        // 製品名
  brand: string;       // ブランド名
  category: "desk" | "studio" | "audio-smart";
  url: string;         // 外部リンク(https のみ許可)
  image?: string;      // 商品画像URL(https のみ許可)
  note?: string;       // ひとことコメント
}

// faq.json
interface FaqData {
  categories: FaqCategory[];
}
interface FaqCategory {
  title: string;
  items: { question: string; answer: string; link?: { label: string; url: string } }[];
}
```

- パーサ(`parseGearData` / `parseFaqData`)は `parseVideosData` と同じ方針で、
  不正データは具体的なメッセージ付きで throw する(ビルドを落として混入を検知)。
- `url` と `image` は `https:` で始まることを検証する(属性注入・javascript: スキーム防止)。

## 7. 詳細設計(主な仕様)

### 7.1 SiteHeader

- `Astro.url.pathname` から現在ページを判定し `aria-current="page"` を付与。
- `<header>` 直下に `<nav aria-label="サイト内ナビゲーション">`。sticky + 背景色 + 下線ボーダー。
- トップページのヒーローと視覚的に干渉しないよう高さは最小限(py-3)。

### 7.2 videos.astro

1. `parseVideosData` で全動画を取得(公開日時降順は取得時に整列済み)。
2. `categorizeVideo` で各動画に主カテゴリを付与。
3. カテゴリチップは動画が 1 件以上あるカテゴリのみ表示(空カテゴリのチップを出さない)。
4. フィルタスクリプトは `aria-pressed` の更新 → `hidden` 切替 → 件数の `aria-live` 更新の順に処理。
5. Shorts バッジ: `isShort === true` のカードに表示。`isShort === null` は横動画扱い(既存方針)。

### 7.3 gear.astro / faq.astro

- JSON をパーサ経由で読み込み、カテゴリ順定義(`GEAR_CATEGORY_ORDER` 等)に従って出力。
- FAQPage JSON-LD の `Answer.text` はリンクを含まないプレーンテキスト
  (表示側のみ `link` を補足リンクとして出す)。
- メールアドレスはスクレイピング対策として概要欄と同じ「☆→@」表記で掲載する。

### 7.4 index.astro / Base.astro の変更

- `Base.astro` の `<body>` に `SiteHeader` / `<slot />` / `SiteFooter` を配置。
- `index.astro` の重複フッターを削除。ヒーロー(`<header>`)はそのまま。
- スキップリンク(`#main`)は既存のまま。各ページの `<main id="main">` を維持する。

## 10. 追補: 全動画アーカイブ対応(PR-B)

初期版は RSS 由来のため videos.json に最新 15 件しか入らず、動画ライブラリも 15 本しか
辿れなかった。全動画をたどれるように 2 点を追加する。

### 10.1 データ経路の二系統化

- RSS は最新 15 件しか返さない仕様上の上限がある。全動画取得には
  **YouTube Data API v3 `playlistItems.list`** で uploads プレイリスト(channelId の
  `UC` を `UU` に置換した ID)を全ページ辿る。
- `scripts/fetch-videos.ts` は `YOUTUBE_API_KEY` があれば API 経路、無ければ RSS 経路を使う。
  API 取得失敗時は RSS にフォールバックし、全経路失敗でも既存データを維持する(ビルドは落とさない)。
- パース関数 `uploadsPlaylistId` / `parsePlaylistItemsPage` は純粋関数として `youtube.ts` に置き、
  RSS の `parseFeed` と同じ `FeedEntry` 形に正規化することで `mergeVideos` 以降を共通化・テスト可能にする。
- API のレスポンスは `contentDetails.videoPublishedAt`(実際の公開日時)を優先し、
  非公開・削除済みで videoId や公開日時を欠くアイテムはスキップする。

### 10.2 段階表示 UI(「もっと見る」)

- 全動画を辿れるようにしつつ初期描画を軽く保つため、`/videos/` は初期 12 件表示とし、
  超過分は「もっと見る」ボタンで展開する(一度展開したら開いたまま・ボタンは再表示しない)。
- カテゴリ絞り込み・キーワード検索と協調し、一致総数で頭打ち判定を行う。
- JS 無効時はボタンを `hidden` のままにして全件表示にフォールバックする(プログレッシブエンハンスメント)。
- 展開時は最初に現れたカードへフォーカスを移してキーボード操作の連続性を保つ。
